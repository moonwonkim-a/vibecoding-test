# AGENT-TEST-WORKFLOW.md

이 문서는 테스트 자동화 에이전트(Claude Code 등)가 따라야 하는 워크플로우 정의다.
에이전트는 매 실행마다 이 문서의 절차를 순서대로 수행한다.

## 0. 입력 문서

| 문서 | 역할 | 에이전트 권한 |
|---|---|---|
| `test-plan.md` | 테스트 정의 (TC ID, 요구사항 ID, 기대 결과) | 읽기 전용 |
| `test-results/INDEX.md` | 실행당 1줄 요약 추이 + 미해결 사항 | 읽기 + 갱신 |
| `test-results/YYYY-MM.md` | 월별 상세 엔트리 | 읽기 + 추가 기록 |
| `database-schema.md` | 목데이터 정책 | 읽기 전용 |

**컨텍스트 절약 규칙**: 에이전트는 매 실행 전 `INDEX.md` 전체와 **직전 Run의 상세 엔트리 1개**만 읽는다.
과거 월별 파일 전체를 읽지 않는다. 과거 상세가 필요하면 INDEX의 링크로 해당 Run만 찾아 읽는다.

## 1. 실행 전 단계 (필수)

1. `test-results/INDEX.md`를 읽는다. "미해결 사항" 목록과 직전 Run의 실패 TC를 확인한다.
2. INDEX 링크를 따라 **직전 Run의 상세 엔트리만** 읽고 "다음 액션" 체크박스를 추출한다.
3. 각 액션 아이템에 대해 현재 코드/환경에서 반영되었는지 검증한다.
   - 반영됨 → 이번 실행에서 해당 TC를 우선 재검증 대상으로 표시
   - 미반영 → 결과 문서에 "이월(carried over)"로 기록하고 사용자에게 보고
4. 직전 실행에서 실패한 TC 목록을 만든다. 이번 실행은 이 목록을 **가장 먼저** 실행한다 (회귀 우선 검증).

## 2. 테스트 실행 단계

1. MCP 서버 연결 상태를 확인한다 (헬스체크 도구가 있으면 호출).
   - 연결 실패 시: 테스트를 실행하지 않고 결과 문서에 "전체 Blocked" 엔트리를 기록 후 종료.
2. `test-plan.md`의 TC를 다음 순서로 실행한다:
   1. 직전 실패 TC (회귀 검증)
   2. 직전 실행 이후 변경된 코드와 연관된 TC
   3. 나머지 전체 TC
3. 각 TC에 대해 기록할 항목:
   - TC ID, 요구사항 ID
   - 결과: PASS / FAIL / BLOCKED / SKIPPED
   - FAIL인 경우: 실제 응답(상태 코드, 바디 요약), 기대 결과와의 차이
4. 실행 규칙:
   - 목데이터는 `database-schema.md` 정책을 따르고, TC 간 데이터 오염을 막기 위해 각 TC 전후 상태를 초기화한다.
   - 실제 개인정보를 절대 사용하지 않는다. `user_code7`은 테스트용 7자리 코드만 사용한다.
   - 동시성 TC(TC-RENT-013)는 MockMvc가 아닌 실제 HTTP 병렬 요청으로 검증한다.
   - 하나의 TC 실패가 다른 TC 실행을 막지 않도록 독립적으로 실행한다.

## 3. 결과 기록 단계

1. **상세 기록**: 현재 월에 해당하는 `test-results/YYYY-MM.md`에 기록한다.
   - 파일이 없으면(월이 바뀌었으면) 템플릿 헤더와 함께 새로 생성한다 (자동 로테이션).
   - `<!-- AGENT:APPEND-BELOW -->` 마커 바로 아래에 `## Run #N — YYYY-MM-DD {#run-N}` 엔트리를 삽입한다 (최신순).
   - 엔트리 구성: 실행 정보 표(시각/환경/커밋/범위/결과) → 실패 상세 → 자기 피드백.
2. **INDEX 갱신**: `test-results/INDEX.md`의 "실행 추이" 표 맨 위에 1줄을 추가한다.
   - 형식: Run 번호 | 날짜 | 통과/전체(통과율) | 실패 TC 나열 | 상세 링크
   - "미해결 사항" 목록을 갱신한다: 해결된 항목은 제거, 새 이월/에스컬레이션은 추가.
3. **비대화 방지 규칙** (필수):
   - 통과한 TC는 상세 엔트리에 나열하지 않는다. 결과는 집계 숫자로만 기록한다.
   - 실패 상세는 실패/차단/직전 대비 변화가 있는 TC만 기록한다.
   - 동일 TC의 반복 실패는 전체 분석을 다시 쓰지 않고 "Run #N-1과 동일 증상 + 새로 알게 된 것"만 기록한다.
   - 응답 바디는 전문을 붙이지 않고 판단에 필요한 부분만 요약한다 (10줄 이내).
4. 기존 엔트리는 절대 수정·삭제하지 않는다. 단, INDEX의 "미해결 사항"은 상태 관리를 위해 갱신한다.

## 4. 자기 피드백 규칙

자기 피드백은 형식적 요약이 아니라 **다음 실행을 개선하는 내용**이어야 한다.

반드시 포함:
1. **잘된 점**: 이전 피드백의 액션 아이템 중 해결된 것. "Run #N-1의 액션 아이템 X → 해결됨" 형태로 명시.
2. **문제점 및 개선 방향**:
   - 반복 실패 TC (2회 이상 연속 실패)는 별도로 표시하고 원인 가설을 갱신한다. 같은 가설을 반복하지 않는다 — 이전 가설이 틀렸다면 무엇을 새로 알았는지 쓴다.
   - 테스트 자체의 결함(플레이키, 잘못된 픽스처, 잘못된 기대값)과 제품 코드 결함을 구분한다.
3. **다음 실행 시 액션 아이템**: 체크박스 목록. 각 아이템은 검증 가능해야 한다 ("개선한다" ❌ / "TC-RENT-013을 RANDOM_PORT 방식으로 전환 후 재실행" ⭕).

피드백 작성 시 금지:
- 결과 표 내용을 그대로 반복하는 것
- 모든 실행에서 복붙 가능한 일반론 ("테스트 커버리지를 높이자" 등)

## 5. 종료 조건 및 에스컬레이션

- 동일 TC가 **3회 연속 실패**하면: 자동 재시도를 멈추고 상세 엔트리에 `🚨 ESCALATION`을 기록한 뒤 INDEX의 "미해결 사항"에 등록해 사람의 판단을 요청한다.
- 통과율이 직전 실행보다 **10%p 이상 하락**하면: 광범위 회귀 가능성으로 보고, 최근 코드 변경 사항을 결과 문서에 함께 기록한다.
- 전체 통과(100%) 시: 액션 아이템에 "프론트엔드 확인 항목(12절) 진행 가능" 을 추가한다.

## 6. MCP 서버 연결 (Cursor 기준)

프로젝트 루트에 `.cursor/mcp.json` 파일을 생성한다 (이 프로젝트에서만 사용).
모든 프로젝트에서 쓰려면 `~/.cursor/mcp.json`에 작성한다. 같은 서버가 양쪽에 있으면 프로젝트 설정이 우선한다.

```json
{
  "mcpServers": {
    "test-runner": {
      "command": "node",
      "args": ["./mcp/test-runner-server.js"],
      "env": {
        "API_BASE_URL": "http://localhost:8080"
      }
    }
  }
}
```

Cursor 측 확인 사항:
- Cursor Settings → Tools & MCP에서 서버가 연결됨(녹색 표시)인지 확인한다. MCP 토글이 꺼져 있으면 켜고 Cursor를 재시작한다.
- 연결 문제 시 Output 패널(Ctrl/Cmd+Shift+U) → MCP 로그를 확인한다.
- Cursor는 전체 MCP 도구 수가 약 40개를 넘으면 일부 도구를 에이전트가 인식하지 못하므로, 테스트 러너 서버의 도구는 5~10개 이내로 유지한다.
- 에이전트가 이 워크플로우를 자동으로 따르도록 `.cursor/rules/test-automation.mdc` 룰 파일을 함께 사용한다 (제공된 파일 참조).

테스트 프로그램 MCP 서버가 제공해야 할 최소 도구(tool) 권장 목록:

| 도구 | 역할 |
|---|---|
| `health_check` | 대상 API 서버 및 DB 연결 확인 |
| `reset_fixtures` | 목데이터 초기화 (database-schema.md 정책) |
| `run_test` | 단일 TC 실행 (입력: TC ID 또는 요청 스펙, 출력: 상태코드/바디) |
| `run_suite` | 영역 단위 실행 (예: RENT 전체) |
| `run_concurrent` | 동시 요청 N건 병렬 실행 (TC-RENT-013용) |

MCP 서버가 없다면 대안: 에이전트가 직접 `./gradlew test` 또는 `mvn test`를 실행하고
JUnit XML 리포트(`build/test-results/test/*.xml`)를 파싱해 동일한 형식으로 기록한다.
